\section{Hipótesis y supuestos}

Para la realización del trabajo práctico se tomaron las siguientes hipótesis:
\begin{itemize}
%	\item Cada producto puede aplicar sólo para una oferta. 
	\item Cada oferta genera un descuento monetario sobre el valor final de la venta.
	\item Existe una única sucursal, con una única caja.
	\item No se persiste ninguno de los datos generados durante la ejecución del programa.
\end{itemize}

\section{Ofertas}

\subsection{Tipos}
Las ofertas fueron clasificadas en 3 categorías:
\begin{itemize}
	\item Por unidad, ya sea marca, categoría o producto.
	\item Por volumen.
	\item Por medio de pago.
\end{itemize}

\subsection{Generación de ofertas}
La casa central del hypermercado, que es la generadora de ofertas, las envía a cada sucursal para su aplicación. Dado que la modelización de dicho comportamiento no es parte del alcance del presente trabajo práctico, las ofertas están representadas en archivos de texto. Para el agregado de nuevas ofertas se deben modificar dichos archivos, que son cargados periódicamente durante la ejecución del programa.

Por cada tipo de oferta existe un archivo distinto.

\subsection{Aplicación de ofertas}
Las condiciones que se deben cumplir para poder aplicar las ofertas en cada caso son diferentes. Debido a la necesidad de variar el comportamiento se implemento el patrón $Strategy$.

En las ofertas, el comportamiento que varia es la decisión sobre si la oferta puede o no aplicarse a un producto. Entonces, cada oferta tiene una colección de criterios de selección de productos (por marca, por categoria de producto, etc). Para saber si una oferta aplica, se debe aplicar el metodo $encajaEnOferta$ de cada uno de los criterios sobre el producto. Éste devuelve un valor lógico que se debe comparar con todos los criterios.

\section{Extensión del sistema}
Tras el objetivo de cumplir con el principio de una implementación clausurada ante cambios y abierta a extensiones, se desarrolló una jerárquia de ofertas en conjunto con la aplicación de patrón $Strategy$. 
Se buscó que la aparición de nuevos tipos de ofertas pueda ser tratada agregando las clases correspondientes, y sin modificar las preexistentes.  

Para crear una nueva oferta, se debe crear un objeto OfertaIndividual, y agregarle los criterios que lo definen. Los criterios son, por defecto, inclusivos, es decir, representan una condición que debe cumplirse. Pero pueden definirse criterios  \textit{excluyentes}, o sea, los productos que coincidan, no entran en la oferta.

A continuación damos unos ejemplos:

\begin{itemize}
	\item Supongamos que queremos una nueva oferta del 10\% en jugos \textit{Rinde dos}. Al objeto OfertaIndividual, le agregamos un criterio por marca, para la marca \textit{Rinde dos}, y le establecemos un valor del 10\%.
	\item Ahora queremos una oferta para todas las bebidas, del 15\%. Creamos una oferta individual, y le agregamos un criterio por categoria, de la categoria \textit{Bebidas}.
	\item Para un descuento del 12\% en bebidas, \textit{excepto} la marca Chandon, debemos crear una oferta individual, agregarle el criterio de categoria ''bebidas'', y agregarle un criterio de marca ''Chandon'', excepto que a éste criterio debemos setearle la condición de \textit{excluyente}.
	\item Queremos una oferta de 2x1 en bebidas. Simplemente agregamos un criterio de selección de productos por categoría, a una oferta por volumen.
\end{itemize}

\section{Aplicación de múltiples ofertas}

Una de las restricciones del trabajo práctico es la de no aplicar más de una oferta a un mismo producto si ya se aplico una oferta del mismo tipo, es decir, sobre un producto no se pueden aplicar dos ofertas 2x1 al mismo tiempo (sino, al comprar 4, se pagaría uno solo).

Sin embargo, sí debe ser posible aplicar ofertas de distinto tipo, por ejemplo, el ejemplo anterior de oferta 2x1 no impide aplicar sobre ése mismo producto una oferta por categoría ''Bebidas''. Entonces, ¿como diferenciar unas de otras?

Para resolver esto, cada clase de oferta tiene un metodo que determina si un descuento es del mismo tipo que ésta genera, es decir, la Oferta Por Volumen reconoce Descuentos Por Volumen, para el resto, ésta función devuelve False. Luego, al aplicar una oferta sobre un Producto Vendido, éste comprueba no tener aplicado ya un descuento del tipo de descuentos que generan la oferta que se quiere aplicar, lo cual indicaría que ya se aplico una oferta similar, y por lo tanto la aplicación de la oferta no es válida.

Por ejemplo, al aplicar una Oferta Por Unidad, el Producto Vendido le pide a la oferta comprobar si alguno de los descuentos que tiene aplicados no es del tipo que éste genera, o sea, Descuentos Por Unidad.

Esto implica que, cuando hay dos ofertas del mismo tipo sobre un mismo producto, \textbf{la primera que se pudo aplicar} será la que finalmente determinará el precio a pagar.
